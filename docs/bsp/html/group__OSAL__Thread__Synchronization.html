◆ R_OSAL_ThsyncCondWaitForTimePeriod()

e_osal_return_t R_OSAL_ThsyncCondWaitForTimePeriod ( osal_cond_handle_t  cond_handle,
osal_mutex_handle_t  mutex_handle,
osal_milli_sec_t  time_period 
)

Wait on a condition variable with timeout for specified time.
This function blocks the calling thread on the condition variable and unlocks the associated mutex.
The calling thread is blocked until either it is unblocked from other thread or the period time has passed.
Return from this function, the mutex is re-locked and owned by the calling thread.

Parameters
[in]cond_handleThe handle acquired at R_OSAL_ThsyncCondCreate().
[in]mutex_handleThe handle acquired at R_OSAL_ThsyncMutexCreate().
[in]time_periodSets the value of time to wait for condition variable unblocked, in msec. (valid values are >= 0)
Returns
e_osal_return_t
Return values
OSAL_RETURN_OKAll arguments are valid, and the condition variable was waited successfully.
OSAL_RETURN_PARInvalid argument.
  • p_time_stamp is NULL.
  • p_time_stamp->tv_sec is out of range. please see also st_osal_time_t.
  • p_time_stamp->tv_nsec is out of range. please see also st_osal_time_t.
OSAL_RETURN_TIMENot unblocked within the specified time.
OSAL_RETURN_FAILAn error occurred in OS or related API.
OSAL_RETURN_HANDLEInvalid handle.
  • cond_handle was not acquired by R_OSAL_ThsyncCondCreate().
  • mutex_handle was not acquired by R_OSAL_ThsyncMutexCreate().
  • cond_handle was already destroyed by R_OSAL_ThsyncCondDestroy().
  • mutex_handle was already destroyed by R_OSAL_ThsyncMutexDestroy().
OSAL_RETURN_STATENot initialized. (Before calling R_OSAL_Initialize().)
Note
This function works on the assumption that the mutex is locked before it is called. The waiting time may be longer than specified due to the influence of other threads.

◆ R_OSAL_ThsyncCondWaitUntilTimeStamp()

e_osal_return_t R_OSAL_ThsyncCondWaitUntilTimeStamp ( osal_cond_handle_t  cond_handle,
osal_mutex_handle_t  mutex_handle,
const st_osal_time_t *const  p_time_stamp 
)

Wait on a condition variable with timeout until time stamp.
This function blocks the calling thread on the condition variable and unlocks the associated mutex.
The calling thread is blocked until either unblocked from other thread or the time stamp is reached.
The value of time stamp should be acquired by R_OSAL_ClockTimeGetTimeStamp().
Return from this function, the mutex is re-locked and owned by the calling thread.

Parameters
[in]cond_handleThe handle acquired at R_OSAL_ThsyncCondCreate().
[in]mutex_handleThe handle acquired at R_OSAL_ThsyncMutexCreate().
[in]p_time_stampSet the pointer address of st_osal_time_t as timestamp that wait until condition variable unblocked.
Returns
e_osal_return_t
Return values
OSAL_RETURN_OKAll arguments are valid, and the condition variable was waited successfully.
OSAL_RETURN_PARInvalid argument.
  • time_period is less than 0.
OSAL_RETURN_TIMENot unblocked within the specified time.
OSAL_RETURN_FAILAn error occurred in OS or related API.
OSAL_RETURN_HANDLEInvalid handle.
  • cond_handle was not acquired by R_OSAL_ThsyncCondCreate().
  • mutex_handle was not acquired by R_OSAL_ThsyncMutexCreate().
  • cond_handle was already destroyed by R_OSAL_ThsyncCondDestroy().
  • mutex_handle was already destroyed by R_OSAL_ThsyncMutexDestroy().
OSAL_RETURN_STATENot initialized. (Before calling R_OSAL_Initialize().)
Note
This function works on the assumption that the mutex is locked before it is called. The minimum unit of time is 1 milli second. Digits less than 1 milli second is rounded up to 1 milli second. If the time stamp is past time, use the current time stamp. The waiting time may be longer than specified due to the influence of other threads.

◆ R_OSAL_ThsyncMemoryBarrier()

e_osal_return_t R_OSAL_ThsyncMemoryBarrier ( void  )

The OSAL memory barrier guarantee that operations called before the barrier are executed before operations after the barrier are executed.

Returns
e_osal_return_t
Return values
OSAL_RETURN_OKThe memory barrier was issued successfully.
OSAL_RETURN_FAILAn error occurred in OS or related API.
OSAL_RETURN_STATENot initialized. (Before calling R_OSAL_Initialize().)
Note
When the memory is shared with multiple devices, such as other CPUs in a multiprocessor system, or memory-mapped peripherals, out-of-order execution may affect program behavior.
For example, a second CPU may see memory changes made by the first CPU in a sequence which differs from program order. This applies to two or more software threads running in a single process whereby these multiple software threads share a single memory space.
Multiple software threads, within a single process, may run concurrently in a multi-core processor environment. Together with out-of-order execution this may lead to unpredictable behavior of the system.

◆ R_OSAL_ThsyncMutexLockForTimePeriod()

e_osal_return_t R_OSAL_ThsyncMutexLockForTimePeriod ( osal_mutex_handle_t  handle,
osal_milli_sec_t  time_period 
)

Lock a mutex assigned to handle with timeout for specified time.
The function blocks the current thread until the mutex assigned to mutex handle is locked by the current thread or until the time period has been elapsed.

Parameters
[in]handleThe handle acquired at R_OSAL_ThsyncMutexCreate().
[in]time_periodSets the value of time to wait for the mutex is unlocked, in msec. (valid values are >= 0)
Returns
e_osal_return_t
Return values
OSAL_RETURN_OKAll arguments are valid, and the mutex was locked successfully.
OSAL_RETURN_PARInvalid argument.
  • time_period is less than 0.
OSAL_RETURN_TIMECould not lock within the specified time.
OSAL_RETURN_FAILAn error occurred in OS or related API.
OSAL_RETURN_BUSYFailed to lock Mutex. This error occurs only in an interrupt context.
OSAL_RETURN_HANDLEInvalid handle.
  • handle was not acquired by R_OSAL_ThsyncMutexCreate().
  • handle was already destroyed by R_OSAL_ThsyncMutexDestroy().
OSAL_RETURN_STATENot initialized. (Before calling R_OSAL_Initialize().)
Note
The waiting time may be longer than specified due to the influence of other threads.
In case of the mutex is already locked, the calling thread is blocked until the mutex becomes available.
In case of the mutex cannot be locked, blocking is terminated when the specified time has passed.
